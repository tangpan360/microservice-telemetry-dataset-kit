# Export Online Boutique Run Data

---

### 0) Prerequisite: Make Prometheus / Jaeger reachable locally

Post-run offline export accesses your local Prometheus / Jaeger Query over HTTP. Keep the following port-forwards running in two **separate terminals** (do not exit):

```bash
# Terminal A: Prometheus
kubectl -n monitoring port-forward svc/monitoring-kube-prometheus-prometheus 19090:9090
```

```bash
# Terminal B: Jaeger Query
kubectl -n observability port-forward svc/jaeger-query 16686:16686
```

### 1) Where this fits in the overall workflow

This stage exports Prometheus metrics and Jaeger traces collected during an `Online Boutique` run into the raw `exported` format.

Assumptions:

- You have finished deployment (see `benchmarks/online_boutique/docs/README.md`)
- You have generated schedules (see `pipelines/traffic/README.md`)
- Outputs of this stage are stored under `dataset/processed/<dataset>/exported/run_id=<run_id>/`

Default access (current version):

- Uses localhost by default:
  - `Prometheus`: `http://127.0.0.1:19090`
  - `Jaeger`: `http://127.0.0.1:16686`
- You can override explicitly via:
  - CLI flags `--prom` / `--jaeger`
  - Environment variables `EXPORT_PROM_URL` / `EXPORT_JAEGER_URL`

### 2) ClarkNet collection

This section replays the `ClarkNet` schedule. We recommend a two-stage process:

- Stage A: inject workload using the schedule only (no incremental online export)
- Stage B: after injection finishes, export `Prometheus` / `Jaeger` post-run based on the time window recorded in `run_manifest.json`

The default ClarkNet schedule generated by this repo is:

- `dataset/processed/traffic/ClarkNet-HTTP/schedules/clarknet_users_10s_peak_u1000.csv`

It corresponds to **weekday-only**, compressed **10-day logical traffic**, so the full run uses:

- `10d`

#### 2.1 Smoke (recommended first, ~5 minutes)

```bash
RUN_ID="clarknet-$(date +%Y%m%d_%H%M%S)"
SCENARIO_ID="clarknet-peak-u1000-step10s-smoke5m"
RUN_DIR="dataset/processed/online_boutique_clarknet/exported/run_id=${RUN_ID}"
SCHED="dataset/processed/traffic/ClarkNet-HTTP/schedules/clarknet_users_10s_peak_u1000.csv"

echo "RUN_ID=$RUN_ID"
echo "SCENARIO_ID=$SCENARIO_ID"
echo "RUN_DIR=$RUN_DIR"

export OB_RUN_ID="$RUN_ID"
export OB_SCENARIO_ID="$SCENARIO_ID"
export OB_RUN_ARTIFACT_DIR="$RUN_DIR"

bash benchmarks/online_boutique/loadgen-locust/run_traffic_schedule_10s.sh "$SCHED" 4 5m --web-port 0

python pipelines/export/export_raw_postrun.py --run-dir "$RUN_DIR"
```

#### 2.2 Full (run the full 10-day logical ClarkNet traffic)

```bash
RUN_ID="clarknet-$(date +%Y%m%d_%H%M%S)"
SCENARIO_ID="clarknet-peak-u1000-step10s"
RUN_DIR="dataset/processed/online_boutique_clarknet/exported/run_id=${RUN_ID}"
SCHED="dataset/processed/traffic/ClarkNet-HTTP/schedules/clarknet_users_10s_peak_u1000.csv"

echo "RUN_ID=$RUN_ID"
echo "SCENARIO_ID=$SCENARIO_ID"
echo "RUN_DIR=$RUN_DIR"

export OB_RUN_ID="$RUN_ID"
export OB_SCENARIO_ID="$SCENARIO_ID"
export OB_RUN_ARTIFACT_DIR="$RUN_DIR"

bash benchmarks/online_boutique/loadgen-locust/run_traffic_schedule_10s.sh "$SCHED" 16 10d --web-port 0

python pipelines/export/export_raw_postrun.py --run-dir "$RUN_DIR"
```

Default behavior:

- Read `inject_start_utc` / `inject_end_utc` from `run_manifest.json`
- Add a 5-minute padding window before and after injection
- Export Prometheus in 60-minute chunks
- Export Jaeger in 10-minute outer chunks and 1-minute slices

To export only Prometheus or only Jaeger:

```bash
python pipelines/export/export_raw_postrun.py --run-dir "$RUN_DIR" --only prom
python pipelines/export/export_raw_postrun.py --run-dir "$RUN_DIR" --only jaeger
```

### 4) What you will get after collection

We recommend storing `exported` under:

- `dataset/processed/<dataset>/exported/run_id=<run_id>/`

Examples:

- `dataset/processed/online_boutique_clarknet/exported/run_id=<run_id>/`
- `dataset/processed/online_boutique_nasa/exported/run_id=<run_id>/`

After post-run export, each exported `meta.json` stores:

- time range information
- `run_id`
- `scenario_id`
- Jaeger `hit_limit`

The injection script also saves:

- `run_manifest.json`: run summary
- `injection_events.jsonl`: injection start/end and other event timestamps

### 5) Test Prometheus / Jaeger export only

Prerequisite: Prometheus / Jaeger must be reachable locally (see the port-forwards in section 0).

If you don't want to start a full run yet and only want to validate the exporter scripts, run the following from the repository root:

```bash
# Prometheus metrics for the last 30 minutes
python pipelines/export/export_prom_metrics.py

# Jaeger traces for the last 30 minutes
python pipelines/export/export_jaeger_traces.py --service frontend.ob
```

Default outputs:

- `runs/export_prom_metrics/<timestamp>/`
- `runs/export_jaeger_traces/<timestamp>/`

### 6) Common arguments

The most commonly used ones:

- `--run-id`: run identifier
- `--scenario-id`: scenario label
- `--out-base-dir`: export output base directory
- `--only prom`: export Prometheus only
- `--only jaeger`: export Jaeger only
- `--dry-run`: show what would be exported without executing
- `--run-dir`: run directory used by `export_raw_postrun.py`
- `--pad-before-minutes` / `--pad-after-minutes`: padding minutes added before/after injection for post-run export
- `--prom-chunk-minutes`: Prometheus post-run export chunk size (minutes)
- `--jaeger-chunk-minutes`: Jaeger post-run export outer chunk size (minutes)

Both single export scripts and post-run export scripts work around time windows. Single export scripts also support explicit time arguments:

- `--minutes`
- `--hours`
- `--days`
- `--start`
- `--end`

### 7) Additional notes

`export_prom_metrics.py` exports the following service-level metrics by default:

- service in/out RPS
- service mean / p95 / p99 latency
- 5xx RPS
- error rate / timeout rate
- CPU
- memory
- CPU throttling ratio
- replicas spec / available
- active node count

For `error_rate` / `timeout_rate` / `latency_*`, queries include safeguards for the zero-traffic boundary:

- When there are no errors/timeouts, return `0` per service instead of dropping the series
- When the request denominator is `0`, return `0` to avoid exporting `NaN`

`export_jaeger_traces.py` filters traces by `frontend.ob` by default.  
If you want to list service names currently available in Jaeger, run:

```bash
kubectl -n observability port-forward svc/jaeger-query 16686:16686
curl "http://127.0.0.1:16686/api/services"
```

### 8) Default behavior of incremental export

Default config file:

- `pipelines/export/export_raw_config.json`

In the config, `url=""` means the exporter auto-selects the default access address:

- Use localhost by default

Default state files:

- `runs/export_raw/state/prom_last_end.json`
- `runs/export_raw/state/jaeger_last_end.json`

Current default strategy:

- Prom: export the most recent unexported `10min` each time, `lag=2min`, `overlap=1min`
- Jaeger: export the most recent unexported `5min` each time, internally sliced into `1min` windows, `service=frontend.ob`
